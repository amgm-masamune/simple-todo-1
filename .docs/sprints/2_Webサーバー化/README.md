# Sprint2 Webサーバー化

振り返り時に作成

## 目標

- 開始時、UseCaseまで実装できているシステムをWebサーバーのエンドポイントとして実装する。
- （認証機能はユーザー機能実装後に実装予定。Bearer・JWT・OAuth2.0・OpenID Connectを学習中）

## 結果

- Webサーバー化実装完了。
  - タスク作成 `POST /task`
  - 特定のIDのタスク取得 `GET /task/{id}`
  - 全てのタスク取得 `GET /tasks`
  - 指定した状態のタスク取得 `GET /tasks/status/{status}`
  - タスク更新 `PUT /task/{id}`
  - タスク削除 `DELETE /task/{id}`
- 技術はモダンで利用されており、TypeScriptに対応・Deno対応・Web標準・軽量等の観点からHonoを採用。
- DTO との変換に Zod を活用。リクエスト・レスポンスのデータはすべて Zod で定義し、スキーマに合うことを保証。
- APIテストはインタフェース周りを重点的に実装。
- リファクタリングで以下を修正
  - 「指定しない」の表現は null から `{ type: "unspecified" }` の形に変更
    （null が未指定を表す、というのは自明ではないため・JSONへの指定で特にあやふやになりやすそうなため）
- GitHub Copilot でのプルリク作成と、その際のAIの自動コードレビュー実装

## 学び

### 初めて Zod を利用。活用タイミングや各レイヤのバリデーションの役割に納得できた

- Zod は unknown を TS の型へパースすることが得意。
- ドメイン層でのバリデーションに使うべきではない。
  - ドメイン層では受け取る型は担保されていると考え、改めてその型をチェックするのは冗長。
  - ドメイン層では制約などのロジックに注力すべき。
- インフラ層で、外界の unknown を型に変換するのに便利。
  - スキーマを定義するとその型を生成してくれる。
  - パースすることでその型であることを保証してくれる。

### 初めて Hono を利用。書きやすいテスト・Web標準の開発体験が良かった

- Web標準（Request等）に則る近年の風潮に沿っている。
- 今までクライアントサイドや、Deno の Fresh 等を利用しており、Web標準の利用はとっつきやすかった。
- テストでは1つのプログラムの中でクライアントとサーバー両方が実行されており、1つのデバッグでどちらのブレイクポイントにも引っかかる体験が面白かった。
  APIも fetch と似ており（request(url, options)・await resp.json() 等）追加の学習コストがほぼなかった。
- Bearer・JWT等もミドルウェアで簡単に実装できると聞くので、今後利用してみたい。

### 型に厳密になりすぎてしまい、型パズルを解くのに時間を掛けてしまった

- ResponseSchema ← SuccessResponseSchema、ErrorResponseSchema のように汎用的に型を定義しようとしたが、
  ZodSchema自体はTSの型ではなくあくまで値なため、汎化が難しい。また、値を取り出すのにも自分の型パズルへの知識不足から
  実装するのに時間がかかってしまった。
- 結局、コードとしても繁雑なものになってしまったため、もっとシンプルにし、目検で正しいと分かる & テストで担保する方がよさそう。

### UseCase のテストと API のテストとの役割分担に納得できた

- UseCase のテスト：型で縛られた中でのあらゆる引数についてテスト。
- APIのテスト：型にとらわれないあらゆる引数についてテスト。
  リクエストは unknown 型でありどんな値でも指定はできるため、特にエッジケースの確認が重要そう。

### リファクタリングに時間を掛け過ぎてしまった

- 気になるところはどんどん直していくフェーズなので、無制限にやってしまいがち。
- 構造の変更などは全体に影響する大きな変更なため、手作業で行うと時間がかかる。AIを活用できる環境ならぜひ使いたい。

### プルリクの大きさを見て、開発を一度に進めようとしていたことに実感した

- feature からさらに task に切る。そこでは動かないコード（not implemented）も含めててよい。
- こまめにプルリクを出し、フィードバックサイクルを高速に回す。
- 自分がどこが完璧主義なのか、コード行数の数字で見て取れた。

## 今後

### 開発する際のブランチをタスク単位で詳細化する

- main・develop → feature/xxx → task/xxx-yyy のように、タスク単位でブランチを切る。
- 各タスクが実装できたらプルリクを作成、AIコードレビューを受ける。
- レビューするタスク粒度を細かく、フィードバックサイクルを高速に回す。

### URL が REST にふさわしくなるよう修正

- 今回の実装では、1件のタスクを扱うAPI（作成・更新・1件取得・削除）は task、複数のタスクを扱うAPIは tasks として定義してしまった。
- RESTでは Task のリソースを表すのは `/tasks` のように複数形にするのが一般的らしい。
