# 学び

## テスト

### 途中振り返り・反省

- テストに掛かる時間が多く、実装に充てられる時間が少ない。
- 最低限を見極める。
  - TDDでのテストは **品質を担保するのが目的ではない**
- 別のテストで担保されているところでも、テストを行ってしまっている。品質を担保する場合はやってもよいが、個人開発の初期としては重く感じる。
  - 例：ドメインで startedAt > completedAt とならないことを担保しているが、DTOとの変換テストでも、APIの入力テストでも行いたくなってしまう。
  - 最低限、バリデーションが動いていることを確認すれば良さそう
- テスト実装が未完了のところは throw "TODO" としたのは良かった。

### ネクストアクション

- 各テスト対象に対するテストは最大で2つに絞る。
  - 4月中に完成させることを目標とするため。
  - 実装するのは理想系テストケース1つ・逸脱系テストケース1（または正常系・異常系）のみにする。
    - 理想系：TDDの目的である、書き方を矯正するため。
    - 逸脱系：最低限バリデーションが動いていることを確認するため。
- 気づいたテストがあれば、その場ではコメントで箇条書きにでも書いておく。

### ネクストアクション後振り返り

- 実装がとても速くなった（10分で1機能など）。

## Git

### 途中振り返り・反省

- 細かくコミットができない。まだ途中だし・まだ修正があるかもだし、と思ってしまうと、どのタイミングでコミットすべきか分からず結果コミットが大きくなる。

### ネクストアクション

- まずは区切りをつける。
  - 最初のテストは2つのみ → ○○のTDDテストを実装
  - 対象の実装 → ○○を実装
  - リファクタリング → ○○をリファクタリング
  「完了」という言葉を付けない。完璧に完了したことを連想してしまう。完了することはない。


## レイヤー

### 振り返り

- Zodを使うことで、Unknownな入力をTSの型に変換するのが楽にできた（リテラル型のユニオンもサポートしている）。
  - TSは型があっていなくても実行できてしまうため、ドメイン層でのunknownバリデーションをすべきか悩んでいたが、
    Zod(インフラ層) が外界からのUnknownな入力をTSの型に変換することを担保するようにすれば、
    ドメイン層では最低限その型で入力されることが保証されていると考えて良さそう。

```
  外の世界
  [[unknown(JSON・入力文字列)]]  --Zodパーサーのフィルタ->  [[DTO(TSの型)]]  --ドメインルールのフィルタ->  [[ドメインエンティティ]]
```

## プルリク・フィードバックサイクル

### 振り返り・反省

- 1ブランチ=1プルリク の意識が無く、1ブランチで1機能全体を完成させ、その全体に対してプルリクを出すイメージだった。
  - 今まで経験した WBS も、実装 → コードレビュー の形になっていたと思う。それはコード全体が完成した後にレビューをするという意味ではなさそう。
  - 結果、1プルリクで**3,000行**になってしまった。
  - また、この状態だと開発の途中にフィードバックをもらえない。
- AIが `router-setup`・`usecase-binding` のような案も提案してきた。
  - 定義のみを実装（そこでは `c.json("not implemented")`）、後でそれを修正して UseCase と結びつける。
  - not implemented のあるコードでプルリクを出す・ブランチを終わらせるという考えがなかった。一気に開発しなくても良い。
- 自分がどこが完璧主義になっているのか自覚できた。
  - なるべく指摘が無い状態で提出する、ということを目指してしまっていた。
  - 結果、プルリクを出す期間も空き、行数も多くなってしまった。
  - そもそもタスクの粒度が大きいため、何に手を付ければいいのか分からないときもあった。

### ネクストアクション

- feature ブランチからさらに task ブランチを切り、完成していないコードも含めてこまめなプルリクを出す。
  - `task/hono-introduction`・`task/router-setup`・`task/usecase-binding` のような案も取り入れる。
  - その中でTDDで実装する。
- 完璧なコードを一気に書こうとするのではなく、少しずつ書いてフィードバックをもらう。

## 疑問点

### Result型を採用すべきか

- TypeScript では throws が無い。
  - interface IRepository で、見つからないときに NotFoundError を投げることを強制できない。
- assertThrows・assertRejects でのエラー内容の指定は Error クラスレベルまたは message 内の文字列レベル
  - オブジェクトレベルで比較するにはテストコード内で try-catch してassertEquals(e.code, "NOT_FOUND") のようにする必要がある。
