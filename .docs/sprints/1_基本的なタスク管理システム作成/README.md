# Sprint1 - 基本的なタスク管理システム作成

## 目標

- タスク管理システムの基本的な機能を実装する。

## 結果

- 基本的な動作確認をCLIで行おうと思ったが、実装に想定以上に時間がかかる見込みなため、Webサーバー化を優先することにした。

## 作ったもの

- タイトル・期限・状態（未着手・進行中・完了・キャンセル）とその日付を管理する。
- メタデータとして ID・システム上での作成日時・更新日時 を持つ。

## 学び

### アジャイル

#### まずは動くものを作る

- 仕様を決め着る前に一旦動くものを作る。

### DDD

#### ドメイン層では型でぎちぎちに固めすぎない

- コンパイルを通すために、呼び出し元が呼び出し方に気を付ける必要が発生、ドメイン知識がユースケースに漏れ出す。

#### ドメイン層は最後の砦。エラーはドメイン層が出す

- 無意識で、アプリ全体として例外が発生しないようにしなくちゃと思っていた。
  - しかし、無効な入力を制御するのはユーザーとの直接のやり取り（UI）である。
  - ドメイン層は、無効な状態を排除する最後の砦。間違っていたら存分にエラーを出してよいし、
    そのエラーが出ないようにユースケースが工夫する必要はない（それはエラーの起きにくいUIの設計の責務）。

### TDD

#### 最初のテストは ユースケースのテスト → ユニットテスト の順で書くべき

- 具体的なユニットはどうなるか分からない。
- ついウォーターフォールのように書きやすいユニットから書いてしまいがち。
- まずはどんなことができるのか、機能の仕様を表現する。

#### 最初のテストは書き方を矯正するため

- テストコードは、品質担保のためのものではない。
- 書き方を矯正するためのもの。

#### 最初のテストがある程度かけたら、あとは作りながらテストを書く

- 素早く開発していく中で、テストを網羅的に書くのは難しい。
- 作りながら、新たに決めた細かい仕様があればテストコードに書いていく。

### クリーンアーキテクチャ・DI・DIP

#### テストがしやすかった

- 抽象に依存できたのは良かった。
- Composition Root は初めてちゃんと書いたが、実装しやすかった。

## 今後

### ドメインは、入力の型は汎用的に、出力は具体的にがよい？

- 今回の学びで、入力は汎用的にしないと呼び出し元にドメイン知識が漏れ出すことがわかった。
- 一方で、利用の際には task.status === "in-progress" ならば task.startedAt は必ず存在する、として扱いたい。
- 入力用のDTOと、返すDomainEntity、という形で扱うのが良いのでは？
