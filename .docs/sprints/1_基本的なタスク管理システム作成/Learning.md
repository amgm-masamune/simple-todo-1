# 学び

## アジャイル

### アジャイルの流れ

1. やりたいことをバックログに積む
2. 今回実現すべきものを選ぶ
3. やるべきタスクを洗い出す
4. 実装する
5. 振り返る

## テスト駆動

### ステップ

1. テストリストで実装すべきテストを書き出す（後程思いついたら追加する）
2. Red：文章で書かれた仕様をテストコードにする
  ※ 構文エラーは無く、実装すればテストコードが通る状態にする
3. Green：テストコードが通るように実装する
4. Refactor：テストコードが通る状態を保ちつつ、コードを綺麗にする


### 最初のテストコード

- 目的：文章で書かれた仕様メモを実行可能な仕様（テストコード）にする。
- Domainを固めるのを最優先にする。
- ドメインルール（状態遷移や制約などの契約）の検証を優先する（コンストラクタなど生成についてはやりすぎない）
- 仕様カバレッジを優先

### テストの書き方

- Given-When-Thenのスタイルで書く
  - Given：前提条件
  - When：操作
  - Then：期待される結果
- 1テスト1アサーションを心がける。テストコード内で分岐は避ける。

## メモ

- テストコード中に 理由・Given-When-Then を書いておくのがよさそう。
  全てではなく重要な個所に絞るのも大切らしい？
- 最初のテストをどのくらい網羅的に書けばいいのか分からない。
- もっと具体的なユーザーフローの仕様を書くべき？ウォーターフォールの考え方で、つい単体レベルのテストから書き始めてしまった
- 内部のクラスとしてはできることがぎちぎちに固まるが、ユーザーとしてはなんなのか、システムの機能としてはなんなのかが定義できてない。そもそも要件定義をちゃんとしてないからか？
  次やるときには要件定義で「タスクは未着手から進行中にすると、先ずはその時の時刻を開始時刻として持つ。あとから変更可能」のような機能を書いてから実装したい。

## 違和感

リファクタリングで改修する可能性があるもの

- テストリスト → 実装 だと、とにかく早く作るという目的を達成できない。あとから修正があったときにテストの修正も必要になる。
  ここでのTDDでのテストはあくまで品質を担保するための物ではなく、書き方を矯正するための物なので、書き方が明確なものは書かなくていいのでは。
- 依存性注入のライブラリを入れてみる（勉強のため）

## 良かったこと

設計等の効果が出たもの

- TaskをほぼValueObject化したことによりバリデーションが統一され安心して書けた。
- テストは事前に書いておくと安心感が違う
