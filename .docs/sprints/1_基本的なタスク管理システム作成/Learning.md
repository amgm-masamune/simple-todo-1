# 学び

## アジャイル

### アジャイルの流れ

1. やりたいことをバックログに積む
2. 今回実現すべきものを選ぶ
3. やるべきタスクを洗い出す
4. 実装する
5. 振り返る

## テスト駆動

### ステップ

1. テストリストで実装すべきテストを書き出す（後程思いついたら追加する）
2. Red：文章で書かれた仕様をテストコードにする
  ※ 構文エラーは無く、実装すればテストコードが通る状態にする
3. Green：テストコードが通るように実装する
4. Refactor：テストコードが通る状態を保ちつつ、コードを綺麗にする


### 最初のテストコード

- 目的：文章で書かれた仕様メモを実行可能な仕様（テストコード）にする。
- Domainを固めるのを最優先にする。
- ドメインルール（状態遷移や制約などの契約）の検証を優先する（コンストラクタなど生成についてはやりすぎない）
- ~~コードカバレッジではなく仕様カバレッジを優先~~
- なるべく「外から見た振る舞い」を書く。
  - 書ききれない詳細があればユニットテストとして書く等。
  - ユニットテストを書きすぎると変更に弱い。
  - **実際にユニットテストから書き始めたので過剰設計気味になっている**。ウォーターフォールとは違う。

### テストの書き方

- Given-When-Thenのスタイルで書く
  - Given：前提条件
  - When：操作
  - Then：期待される結果
- 1テスト1アサーションを心がける。テストコード内で分岐は避ける。

### 開発の流れ

- リファクタリングを行う前に Git にコミットする。（今回やらなかった）
  - 基の状態がどうだったのか、それがどう改善されたのかが分かる。
  - 今までどんなことを試してきたかが分かる。

## DDD

- 実現したいことが不明確だとドメインが固まらない。結果、キャンセルは本当に必要なのか等いらない機能も入れてしまう可能性がある。
- 複雑なドメインを、正しく理解して正しくコードに表現できるようにするもの。
  制約が強くなるので、何にでも適用すべきものではない。
- ユースケースでは受け取った入力をそのままドメインエンティティとしない。
  外から来たものを信用することになる。`type TaskUpdateInput` や `if (task.title === undefined)` 等が必要になる。
  必ずユースケースの中でリポジトリから取得、その場で更新し、保存する。
- **ユースケースにドメインルールが漏れないよう**にする。あくまでも全体のオーケストレーション。

## オブジェクト指向 or 関数型

- 今回、イミュータブルなクラスの形で実装した。
- 両者でできることに大きな差はない。
- ただ、クラスだと実装を複雑に書きすぎてしまう。
- しかし今回はドメインエンティティ・ドメインクラスの実装が凝集されること、また自分の書きやすさより、クラスを選んだ。

## メモ

- テストコード中に 理由・Given-When-Then を書いておくのがよさそう。
  全てではなく重要な個所に絞るのも大切らしい？
- 最初のテストをどのくらい網羅的に書けばいいのか分からない。
- もっと具体的なユーザーフローの仕様を書くべき？ウォーターフォールの考え方で、つい単体レベルのテストから書き始めてしまった
- 要件が内部モデルに寄りすぎ。
  内部のクラスとしてはできることがぎちぎちに固まるが、ユーザーとしてはなんなのか、システムの機能としてはなんなのかが定義できてない。そもそも要件定義をちゃんとしてないからか？
  次やるときには要件定義で「タスクは未着手から進行中にすると、先ずはその時の時刻を開始時刻として持つ。あとから変更可能」のような機能を書いてから実装したい。

## 違和感・リファクタリング予定

- ~~テストリスト → 実装 だと、とにかく早く作るという目的を達成できない。あとから修正があったときにテストの修正も必要になる。~~
  ここでのTDDでのテストはあくまで品質を担保するための物ではなく、書き方を矯正するための物なので、書き方が明確なものは書かなくていいのでは。
  ↑ ユニットテストを重くし過ぎた。タスク作成とかはユースケースのテストとして書くべき
- 依存性注入のライブラリを入れてみる（勉強のため）
- プロパティの型の定義が散在している（プロパティの定義・コンストラクタ・setter・サービス）
- リポジトリの検索系はどこまで詳細化して書くか悩む
- 時間があれば、UUIDはDBのパフォーマンス考えるとv7にしたい
- TaskServiceのServiceという用語は安易に選んでしまったが、アプリケーションサービスとドメインサービスとで紛らわしいのでUseCaseにする。

## 勉強不足

- DDDの境界付けられたコンテキスト。タスクの扱い方等が正しいか分からないので完成後勉強する。

## 良かったこと

設計等の効果が出たもの

- TaskをほぼValueObject化したことによりバリデーションが統一され安心して書けた。
- テストは事前に書いておくと実装がやりやすい。
- モジュールが出来たときにはテストを実行しているので、全部完成して初めて実行、みたいなことが無く着実に作っていける。
- TaskServiceのTDDは、重すぎず実装の指針にもなってよかった（もしかしたら書き慣れていたからかも）
- 依存性注入でリポジトリの依存を注入する形にしたため、実DBとの切り替えが簡単になりそう。
