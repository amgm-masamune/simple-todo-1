# 超絶シンプルなタスクアプリ

[GitHub](https://github.com/amgm-masamune/simple-todo-1)

基本的にAIにコードを書かせず、自分の手で作るタスク管理のWebアプリ(予定)。
とにかく早く実装し、その後気になった部分をリファクタリングしていく。
設計にこだわりすぎないよう、書き方を矯正できるTDDを一部取り入れて実装する。
自分が成功体験を積むための、あくまで自分のためのプロジェクト。

要件は .docs/plan/given-ver1.md にある

- TDD：設計にこだわりすぎないよう書き方を矯正するため
- DDD：実装よりも前に、どのようなドメインを扱うのか、ドメインに対する知識を深めるため
  （今回はタスク管理アプリなので軽量DDDの実践的なところのみ利用するかも）

## 目的

### 主目的

- 一度作り切り、振り返りを基にリファクタリングすることを回す。
  考えすぎ・こだわりすぎて進まないという現在の思考を矯正する。

### サブ目的

- アジャイルの考え方を学ぶ（とにかく早く出す）。
- ~~TDD・DDDの理解を深める。~~
  ↑ 今の段階(とにかく早いものを作る)ではTDDにこだわってしまう原因になってしまう。
    以降、書き方を矯正するため最低限のテストコードのみとする。

## 進め方

1. 現在の理解のまま~~TDD・DDDで~~実装を進め、**完成させる。作り切る**ことが目的。
  - なんか違う・非効率だと思ってもとにかく作り切る。
  - 違和感を感じたら記録しておく。
  - 作り始めたときの物が、実際に作ってみてどうだったのかを振り返るものにする。
2. 完成品をAIに評価してもらう。振り返る。
3. 1で作ったものをリファクタリングする。
4. 品質担保用の単体テスト・結合テスト・総合テストを可能な限り追加する。

## 全体の振り返り

### 目的の再確認

- 自分の手を動かして作ってみる
- アジャイルやTDD、DDD、Git Flow 等の開発手法の実践・自分なりの所感を得る
- こだわりすぎないようにする（とにかく完成させることを優先する）
- 自分が使いなれている技術に加え、良く使われている技術を活用する

### 学び

- Deno
  - 今まで使い慣れていた技術。
  - npm 対応したこともあり、不都合はなかった。
  - シンプルなコンフィグファイルで簡単に実行できる。動作も node に比べて軽量。
- Hono
  - 初めて使ったが、Web標準でなじみやすかった。
  - 認証機能の実装が間に合わなかったのが残念だが、ミドルウェアが用意されていることをきっかけに
    改めて Basic・Bearer認証を学び直し、JWT・OAuth・OpenID Connect などの学習へつなげられた。
  - 一方本システムはアプリのバックエンドの想定だが、HonoやCRUDでよかったのか検討不足だった。
- Zod
  - ドメインロジックの整合性をバリデーションするものではなく、型があっているかを担保するためのものだと理解した。
  - TSの型をほとんど定義できるのが面白かった。
- Drizzle ORM・PostgreSQL
  - TypeScript でスキーマを定義すれば自動でマイグレーションファイルを生成してくれるDXが非常に良かった。
  - 今まで自主的にDBを活用したことが無かった（ファイルやKV等で完結させてしまっていた）。
    気軽に活用できることが分かった。SQLや正規化等は知っていたが、トランザクションを実践したのは初めてだった。
  - 一方、先ず完成させることが前提ならトランザクションは最初は無くてもよかったかもしれない。
  - また、CIでのテスト並列実行時にDBの初期化で苦しめられてしまった。
- GitHub・Git Flow・CI
  - ブランチとプルリクが対応しているという基本的な知識を知らず、プルリクを3000行レベルで作成してしまった。
  - 細かなIssueとブランチ・プルリクを活用することで、目的や意図を集約できそうでチーム開発では有効に活用したい。
  - 一方、個人開発においては Git Flow のブランチ切り替えやプルリク作成が煩わしく感じてしまった。
    個人開発では、ローカルで main に細かくコミット、動くものになったら push してCIで確認でも十分かもしれない。
  - CIはもっと設定が必要かと思っていたが、リポジトリ内に設定ファイルを置いてPushするだけで実行できるのは非常に手軽だった。
- Docker
  - 他プロジェクトとのバージョン競合等を意識せずに環境構築できるのは非常に良かった。
  - ログの確認や環境変数の指定、内部ネットワークなども手を動かして学べた。
- TDD・テスト
  - テストコードを書くことをあまりしてこなかったため、テストに守られている安心感が大きかった。
- 依存性注入
  - テストの際にモックに差し替えるのが容易だった。
  - インスタンス生成を一元管理することで、コードの見通しが良くなった。
- 全体
  - 一人で作り上げたのは数えるほどだが、今までは変なところで悩んで挫折してきたため、
    モダンな技術を活用して作り上げられたのは達成感が大きい。
  - コードや大まかな設計方針は自分で考え実践することができた。不具合の修正方針などはAIを頼りすぎてしまった。

### 反省

- タスク管理アプリ、しかもタスクドメインのみで認証もなく、バックエンドのみという中途半端なものになってしまった。
- テストケース・抽象化を考えすぎてしまい、手が止まってしまうことがあった。
- 最初に書くときに読みやすいコードになるよう意識しすぎてしまい、時間がかかった。
- リファクタリングも考えすぎてしまい、手が止まってしまった。
- コードのリファクタリングの修正作業が煩わしいと感じてしまい、より考えて書くようになってしまいそうだが、
  次のプロダクトではAIを活用するため、先ずは完成させるという目的をより意識したい。
- TDDのテストコードは最低限のつもりだったが、いつの間にかテストを書くことが目的になってしまっていた。
- 開発が遅かったため実際に使ってみての改善、というものが無かった。
- 抽象化しすぎると関数のオーケストレーションばかりになり、コードが塊でなく細切れに見えるため
  返って視認性が悪くなることもあると感じた。
